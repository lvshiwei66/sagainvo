# DOMPDF 高度计算修复报告

## 问题背景
在dompdf.js迁移过程中，我们发现导出的PDF存在多余的空白页问题。最初我们尝试使用高度调整因子（-22.67mm或-64px）来解决这个问题，但这种方法效果有限，且无法根本解决分页问题。

## 根本原因分析
经过深入分析，发现问题的根本原因在于：

1. **固定最小高度设置**：预览组件和导出函数中设置了固定的`minHeight`值，强制容器占据A4页面的大部分空间
2. **DOMPDF分页算法**：DOMPDF无法智能地根据实际内容高度进行调整，而是严格遵守容器的尺寸设定
3. **高度调整因子无效**：无论调整因子如何变化，只要设置了固定的最小高度，就会导致内容无法紧凑排列

## 解决方案
我们采取了以下根本性修复措施：

### 1. 移除固定最小高度
- 将所有`minHeight`设置改为`'unset'`，让内容决定实际高度
- 添加`height: 'auto'`让DOMPDF自动计算实际内容高度

### 2. 修改预览组件
- InvoicePreview.dompdf.tsx 中移除了预设的最小高度
- Notes/Terms 区域的 `minHeight: '60px'` 也被移除

### 3. 修改导出逻辑
- dompdf-export.ts 中临时创建的元素不再设置固定高度
- 现有元素的高度设置也改为动态计算

## 修复内容

### src/lib/dompdf-export.ts
- 移除了 `A4_HEIGHT_ADJUSTMENT_FACTOR_MM` 常量
- 修改了两个地方的样式设置：
  - 现有元素：`minHeight` -> `'unset'`, `height` -> `'auto'`
  - 临时元素：`minHeight` -> `'unset'`, `height` -> `'auto'`

### src/components/invoice/InvoicePreview.dompdf.tsx
- 预览容器：`minHeight` -> `'unset'`
- Notes/Terms 区域：`minHeight: '60px'` -> `minHeight: 'unset'`

## 逻辑原理
现在PDF导出的工作方式是：
1. 不再强制设置最小高度，让内容自然决定所需空间
2. DOMPDF会根据实际内容高度生成PDF
3. 内容紧凑排列，没有多余空白页
4. 只有当内容确实超过A4页面高度时才会分页

## 验证结果
- ✅ 简单发票内容能正确显示在一页上
- ✅ 高度计算基于实际内容而非预设值
- ✅ 消除了多余空白页问题
- ✅ 长内容仍能正确分页
